# 可演示稳态 L3 助手阶段总结

## 1. 本阶段结论

当前版本已经可以作为“可演示的稳态 L3 助手”进行阶段性收尾。

这里的“稳态 L3”指的是：

- 用户可以用中文正常和系统交流
- 系统能在需要时切换到检索、工具、高风险确认、持续执行任务
- 失败、等待、继续执行这些关键状态会用客户能理解的话反馈出来
- 一组典型客户场景已经被手工回归和自动测试覆盖

这不等于生产级平台化成熟，也不等于事实严格核验系统。

## 2. 当前已经稳定下来的能力

### 2.1 正常中文问答

- 普通中文问题可以直接回答
- 对仓库原理类问题，会优先走检索接地而不是误升到 workflow
- 中文回答不再默认带明显系统腔和内部术语

### 2.2 高风险确认

- 对高风险动作，系统会先要求确认
- 确认提示会用自然中文说明“为什么要确认”
- 不会直接执行后再补解释

### 2.3 持续执行任务

- 用户明确要求“继续跟进 / 持续执行”时，系统会进入 workflow_task
- 前端可以展示任务卡片、状态和运行细节
- 任务完成后，用户继续追问“现在进展到哪一步了？”，系统会优先基于最近任务状态回答

### 2.4 失败反馈

- 工具失败时，会返回自然中文解释
- 不会把底层 reason code、内部状态名、原始错误直接甩给客户
- 会尽量给出下一步建议，而不是只提示失败

### 2.5 常见顾问型问题

下面这类问题已经做了稳定回答收口：

- “你能帮我做什么？”
- “这个 workflow runtime 是怎么工作的？”
- “基于当前项目状态，给我一个可落地的优化方案，优先按收益排序。”
- “帮我定位这个仓库里最值得先看的关键模块，并解释它们之间的关系。”

## 3. 已验证的客户场景

这一版已经围绕下面这些真实客户请求做过回归：

- 登录后进入聊天页并发起对话
- 中文仓库原理问答
- 高风险请求的确认
- 确认后转为持续执行任务
- 持续执行任务完成后的进展追问
- 工具失败时的人话反馈
- 优化方案建议
- 模块地图解释

这些场景既做过人工真实回归，也补进了一部分后端契约测试。

## 4. 当前的可靠性边界

这一版已经强调“能不能用”，但还没有扩到更高等级的平台化要求。

当前边界如下：

- 不是严格事实核验系统
  - 目前有检索接地、路由约束、审批治理、prompt 禁止捏造
  - 但还没有做到逐条事实核验、强引用约束、低置信度拒答

- 不是生产级多租户平台
  - 当前更适合本地演示、受控环境和能力展示
  - 还没有完成真正的生产级隔离、治理和运维收口

- 顾问型复杂分析仍有上限
  - 常见问题已做稳定化
  - 但特别开放、特别复杂的方案分析，仍可能退化成较泛的总结

## 5. 为什么现在适合阶段性收尾

本阶段目标不是“无所不能”，而是：

> 做到一个客户第一次使用时，会觉得它能正常交流、能解释、能确认、能失败恢复、能继续执行的助手。

按这个目标判断，当前版本已经达标。

如果继续无限制扩功能，风险反而更大：

- 把已经稳定的交互重新改松
- 目标发散，从“客户能不能用”滑回“工程上还能不能更优雅”
- 破坏已经建立的演示闭环

## 6. 建议冻结的能力边界

当前版本建议对外这样表述：

- 这是一个可演示的 L3 级任务型智能助手
- 它已经支持中文问答、检索接地、高风险确认、持续执行和任务进展追问
- 它强调真实交互和可靠演示，不强调生产级平台化或严格事实核验

不建议对外承诺：

- 完整生产级平台
- 全场景顾问型强分析
- 严格反幻觉事实验证
- 全部工具都已达到企业级稳定性

## 7. 下一阶段如果要继续做

下一阶段不建议继续盲目扩功能，而应该围绕可靠性继续加固：

1. 回答层事实可靠性
   - 给事实型回答增加更强的引用和证据约束

2. 浏览器端端到端契约
   - 把关键演示链路固化成浏览器 E2E，而不只靠手工回归

3. demo 链路和正式链路收口
   - 逐步缩小 demo API 与正式后端运行栈的行为差异

## 8. 推荐对外一句话描述

> 当前版本已经是一个可演示的稳态 L3 助手：它能和用户正常中文交流，在需要时发起持续执行任务，并对确认、失败和进展状态给出自然、可信、可继续操作的反馈。
