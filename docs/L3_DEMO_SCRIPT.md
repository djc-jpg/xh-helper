# L3 助手演示脚本

## 1. 演示目标

这份脚本不是为了展示“功能很多”，而是为了稳定展示三件事：

- 它能像正常助手一样交流
- 它不会在高风险动作上乱执行
- 它可以把一次请求推进成持续执行任务，并能继续承接后续追问

## 2. 推荐演示顺序

### 场景一：正常中文问答

用户输入：

```text
这个 workflow runtime 是怎么工作的？请用中文简洁说明。
```

期待效果：

- 返回正常中文
- 不升级成任务创建
- 回答聚焦仓库里的 workflow runtime，而不是泛化成自我介绍

### 场景二：能力介绍

用户输入：

```text
你能帮我做什么？
```

期待效果：

- 返回自然中文能力介绍
- 不出现明显系统术语堆叠
- 不把内部状态、工具名、实现细节直接砸给用户

### 场景三：高风险确认

用户输入：

```text
send ticket to oncall team
```

期待效果：

- 系统先确认，不直接执行
- 提示语自然，能解释这是高风险动作

### 场景四：确认后继续执行

用户操作：

- 点击“确认并继续”

期待效果：

- 系统进入持续执行任务
- 前端显示任务卡片、状态或运行细节

### 场景五：任务进展追问

用户输入：

```text
现在进展到哪一步了？
```

期待效果：

- 系统基于最近任务状态回答
- 不退回泛化自我介绍
- 能说明任务还在进行中，或者已经完成

### 场景六：失败反馈

用户输入：

```text
search fail temporal workflow docs
```

期待效果：

- 返回自然中文失败提示
- 不直接暴露内部错误码
- 最好给出替代建议或下一步建议

### 场景七：顾问型回答

用户输入：

```text
基于当前项目状态，给我一个可落地的优化方案，优先按收益排序。
```

期待效果：

- 能直接给出几个优先级清楚的建议
- 不退回“我可以帮你回答问题”的泛化介绍

### 场景八：仓库模块地图

用户输入：

```text
帮我定位这个仓库里最值得先看的关键模块，并解释它们之间的关系。
```

期待效果：

- 明确点出 `apps/api`、`apps/worker`、`runtime_backbone`、`apps/frontend`
- 能解释它们之间的关系

## 3. 演示时建议强调的话术

建议强调：

- 这不是单纯聊天壳，它能在需要时进入受控执行链路
- 它不会把高风险动作伪装成普通问答
- 它能把一次对话推进成持续执行任务
- 失败时会说人话，不会把客户扔给内部状态码

不建议强调：

- 它已经是生产级平台
- 它已经具备强事实核验
- 它适合所有开放式顾问分析场景

## 4. 如果现场表现不稳定，优先使用的保守路径

为了保证演示成功率，优先使用下面这些稳定场景：

1. 仓库原理中文问答
2. 高风险确认
3. 持续执行任务
4. 任务进展追问
5. 模块关系解释

尽量避免一开始就使用特别开放、特别长、特别抽象的咨询型请求。

## 5. 演示完成后的收口话术

推荐收口：

> 当前版本的重点不是“无所不能”，而是“在真实任务交互里，像一个可靠助手一样工作”。它已经能正常交流、受控执行、持续跟进，并把关键状态用客户能理解的话反馈出来。
