# MAS `ticket_email` 任务指标目标与完成标准

## 1. 任务类型选择

- 本阶段优先改进的任务类型：`ticket_email`。
- 任务定义：从客服邮件进入系统开始，经 MAS 完成分类、分派、执行与结果回写，直到任务进入终态。
- 目标：先在单一高频任务上实现稳定高质量，再复用同一指标框架到其他任务类型。

## 2. 核心指标与阈值

### 2.1 成功率（Success Rate）

- 指标名：`task_success_rate`
- 计算口径：
  - `task_success_rate = successful_tasks / created_tasks`
  - `successful_tasks`：在 SLA 窗口内完成并进入成功终态的 `ticket_email` 任务数
  - `created_tasks`：同一统计窗口内新创建的 `ticket_email` 任务总数
- 目标阈值：`> 95%`
- 预警阈值：滚动 24 小时 `< 95%`
- 预期达成标准：
  - 滚动 14 天平均值 `> 95%`
  - 同一 14 天窗口内，低于 95% 的天数不超过 1 天

### 2.2 时延（Latency）

- 指标名：`task_latency_ms`
- 计算口径：
  - 端到端时延：任务从创建到完成（`SUCCEEDED` 或业务定义终态）的总耗时
  - 阶段时延：按 MAS 各阶段分别统计（接入/解析、决策/规划、执行/回写）
- 目标阈值：
  - 端到端时延：`p95 < 500ms`
  - 各阶段时延：`p95 < 500ms`（逐阶段独立考核）
- 推荐阶段预算（工程优化参考）：
  - 接入/解析：`<= 100ms`
  - 决策/规划：`<= 200ms`
  - 执行/回写：`<= 200ms`
- 预警阈值：
  - 滚动 1 小时端到端 `p95 >= 500ms`
  - 任一阶段滚动 1 小时 `p95 >= 500ms`
- 预期达成标准：
  - 滚动 7 天端到端 `p95 < 500ms`
  - 任一阶段不出现连续 2 个以上小时窗口 `p95 >= 500ms`

### 2.3 重试率（Retry Rate）

- 指标名：`task_retry_rate`
- 计算口径：
  - `task_retry_rate = tasks_with_error_retry / created_tasks`
  - 仅统计“因系统错误触发的自动重试”，不包含人工 rerun 与业务批准重放
- 目标阈值：`< 5%`
- 预警阈值：滚动 24 小时 `>= 5%`
- 预期达成标准：
  - 滚动 14 天平均值 `< 5%`
  - 同一 14 天窗口内，高于 5% 的天数不超过 1 天

## 3. 完成标准（Definition of Done）

满足以下全部条件，视为本轮 `ticket_email` 指标建设完成：

1. 已形成书面目标文档，明确任务范围与三项核心指标定义。
2. 三项指标均有明确阈值并可核查：
   - 成功率 `> 95%`
   - 时延（端到端与阶段）`p95 < 500ms`
   - 重试率 `< 5%`
3. 监控系统可按任务类型（`ticket_email`）与时间窗口（小时/天/滚动）输出指标。
4. 告警规则已按阈值落地并可触发验证。
5. 达成标准（7 天/14 天滚动）可审计、可复盘。

## 4. 统计与评审节奏

- 统计粒度：
  - 运维监控：按小时
  - 管理评审：按天 + 滚动 7 天/14 天
- 评审节奏：
  - 每日：检查阈值越界与主要错误原因
  - 每周：复盘阶段瓶颈与改进项关闭情况

## 5. 变更说明

- 当业务流量或基础设施发生明显变化时，允许重新标定阈值，但必须记录变更日期、原因与对比数据。
- 在 `ticket_email` 达标并稳定后，按同模板扩展到其他任务类型。
